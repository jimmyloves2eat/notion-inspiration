# Notion 靈感板 Skill

把任何網址轉成 Notion 靈感卡片，自動填好所有欄位。所有同事共用一個 GAS 中繼站，不需要各自設定 Notion Token。

## 密鑰處理（首次使用）

---

🚨 **架構原則（絕對不能違反）**：

這個 Skill **唯一**的 Notion 寫入路徑是透過 GAS 中繼站，**永遠用 GAS**。

即使你看到本機環境變數有 `NOTION_API_TOKEN`、`NOTION_DATABASE_ID`（管理員測試殘留），**也絕對不可以**用它們直接呼叫 Notion API。理由：
- 這個 Skill 是設計給整個團隊用，同事們**不會有** Notion API Token
- 走 GAS 路徑才會跑過所有正確邏輯（檔案上傳、cover 設定、source URL、欄位校驗）
- 直接打 Notion API 會繞過所有 GAS 內建的驗證和處理，產生壞資料

**唯一可用的 Notion 寫入方式**：Step 4 的 Python + curl 呼叫 GAS URL。如果你發現環境變數沒 `NOTION_GAS_SECRET`，就走「## 密鑰處理（首次使用）」流程跟使用者要密鑰，**不要**改去找其他 Notion 憑證。

---


GAS 中繼站網址已寫在 Skill 中（見 Step 4 程式碼），不需要設定。但**團隊共用密鑰**需要使用者在第一次使用時提供一次。

在 Step 4 送出資料之前，**先檢查環境變數 `NOTION_GAS_SECRET`**：

- **如果有值** → 直接拿來用，不要打擾使用者
- **如果沒有值** → 在對話中問使用者：
  ```
  這是你第一次使用這個 Skill。請輸入團隊共用密鑰：
  （如果不知道密鑰，請向設計團隊管理員索取）
  ```
  使用者輸入後：
  1. **本次對話直接拿來用**（不需要重啟 Claude Code）
  2. 詢問是否要存起來：「要把密鑰存進 `.claude/settings.local.json` 嗎？這樣未來就不用再輸入。」
  3. 如果使用者同意，用 Edit 或 Write 工具把以下內容合併進 `.claude/settings.local.json`（注意：合併進 `env` 區塊，不要覆蓋掉既有設定）：
     ```json
     {
       "env": {
         "NOTION_GAS_SECRET": "（使用者剛輸入的密鑰）"
       }
     }
     ```
  4. 告訴使用者：「✅ 已存好，下次起會自動帶入。本次先用你剛輸入的密鑰送出。」

**絕對不要把密鑰寫進對話、log、或其他公開檔案。** 它只應該存在於 `settings.local.json` 的 env 區塊中。

## 資料庫欄位（只有這 6 個，不能自創欄位）

| 欄位名稱 | Notion 屬性類型 | 說明 |
|---------|--------------|------|
| `項目名稱` | title | 頁面標題 |
| `項目種類` | select | 單選，只能用下方列出的選項 |
| `推薦指數` | select | 單選，只能用下方列出的選項 |
| `關鍵字` | multi_select | 多選，只能用下方列出的選項 |
| `說明文字` | rich_text | 品牌視覺分析（200字以內） |
| `檔案和媒體` | files | 封面圖 URL |

### 項目種類（select）— 只能選一個
- `平面設計`
- `標準字`
- `UI / UX`（斜線前後各有一個空格）
- `影片`
- `插畫`
- `AI Prompt`

### 推薦指數（select）— 只能選一個
- `⭐`
- `⭐⭐`
- `⭐⭐⭐`
- `⭐⭐⭐⭐`
- `⭐⭐⭐⭐⭐`

### 關鍵字（multi_select）— 只能從以下選項中選，**最多選 2 個**

**極度重要**：每個選項的斜線前後都有一個半形空格，**不能省略**！如果省略空格，Notion 會在資料庫裡新建一個重複的選項，造成髒資料。請逐字比對。

**選擇規則**：
- **最多 2 個**（嚴格上限）。不是「能選就多選」，是「最能代表主視覺的那 1-2 個」
- 寧缺勿濫：如果一張圖只有 1 個關鍵字能精準描述，就只選 1 個
- **以圖片實際視覺特徵為準**：不要因為「品牌類型」就硬塞（例如：任天堂遊戲視覺不必選 `戰爭/策略`、AI 公司官網不必選 `AI 相關`），但反過來，**如果視覺特徵本身就明顯符合**（例如海報裡角色拿武器），就要選對應關鍵字，不要因為畫面有其他干擾元素（如光效）就跳過

**每個關鍵字的具體判定標準**：

| 關鍵字 | 視覺特徵（看到這些再選） |
|--------|-----------------------|
| `極簡 / 都會` | 大量留白、無襯線字體、灰階低彩度、黑白攝影／配色、幾何構圖、線條設計、扁平化、Clean UI（高對比、無多餘裝飾、清晰資訊層級） |
| `科技 / 現代` | 深色背景／Dark mode、霓虹／螢光色、發光特效（Glow/Bloom）、未來感字型、全息投影感（Holographic）、玻璃擬物化（Glassmorphism） |
| `戰爭 / 策略` | 軍事符號／武器、戰術地圖／棋盤格、HUD 介面、等角透視（Isometric）、嚴肅冷峻配色、金屬／裝甲材質、粗獷主義（Brutalism） |
| `手繪 / 漫畫` | 線稿、漫畫人物（日系為主）、明顯筆觸（Brush strokes）、不規則邊緣、網點（Halftone）、水彩暈染、塗鴉風（Doodle，隨手簡筆畫感）、插畫風格 |
| `可愛 / 童趣` | 圓潤造型／無銳利邊角、粉嫩色、Q 版角色（粗短身形）、無輪廓線色塊構成 |
| `傳統 / 復古` | 東方元素（中式紋飾、書法、水墨、印章）、復古印刷感、舊報紙／褪色感（Sepia）、雜訊與斑駁（Grunge/Noise） |
| `紋理 / 材質` | 紙紋／顆粒感（Grain）、金屬／布料／皮革質感、厚塗顏料肌理、剪紙層次效果、擬物化（Skeuomorphism） |
| `3D / 特效` | 立體渲染（Spline／Blender 感）、景深模糊（Depth of Field）、材質光澤（Glossy/Matte）、黏土人質感（Claymorphism）、流體／氣球膨脹感、次表面散射（SSS／透光感） |
| `潮流 / 誇張` | 字體變形（液態化／3D 擠壓／扭曲／金屬反光）、流行文化梗、街頭文化（街頭塗鴉 Graffiti、滑板、嘻哈、潮牌）、高飽和撞色（紫紅+螢光綠、粉橘+電光藍）、Y2K 美學、低保真／glitch、酸性設計（Acid Graphics）、反骨／解構排版 |
| `AI 相關` | AI 工具截圖、AI 生成藝術、Prompt 範例 |

**注意**：
- `AI 相關`「AI」和「相關」之間有一個空格，不能省略
- 一張圖如果既有 `3D / 特效` 又有 `潮流 / 誇張`，選那個視覺主導性更強的一個
- 不確定時，**少選**，不要多選湊數

**重複詞彙的歸屬判斷**（同一個字在多個關鍵字出現時，看視覺脈絡決定）：

- **金屬**：
  - 出現在武器、裝甲、軍事場景 → `戰爭 / 策略`
  - 單純當作材質紋理模擬（金屬刷紋、霧面金屬底）→ `紋理 / 材質`
  - 用在字體效果（chrome 反光、液態金屬字）→ `潮流 / 誇張`

- **塗鴉**：
  - 隨手簡筆畫感、可愛塗鴉、童趣插畫風（Doodle）→ `手繪 / 漫畫`
  - 街頭噴漆藝術、Hip-hop 文化視覺（Graffiti）→ `潮流 / 誇張`

- **黑色背景**：
  - 配霓虹／螢光、發光特效、未來感、無人物 → `科技 / 現代`
  - 配冷峻軍事色、武器、戰鬥場景 → `戰爭 / 策略`

- **3D 渲染**：
  - 寫實光影、產品攝影感、無角色 → `3D / 特效`
  - 流體變形、誇張造型、Y2K 風 → `潮流 / 誇張`
  - 3D 角色 + 武器／戰鬥姿態 → `戰爭 / 策略`（不要因為是 3D 渲染就選 `3D / 特效`）

- **科幻發光特效**：
  - 出現在角色身上（科幻武器發光、能量罩、戰鬥特效）→ `戰爭 / 策略`（**角色拿武器/戰鬥姿態的訊號永遠優先於光效美學**）
  - 純抽象環境、UI、介面背景的光效 → `科技 / 現代`

- **科幻角色**：
  - 機甲戰士、武裝角色、戰鬥場景 → `戰爭 / 策略`（電競海報、FPS 遊戲視覺等都歸這裡）
  - 抽象人物剪影、無武器、純未來感造型 → `科技 / 現代`

**戰爭 / 策略 的硬性訊號**（只要出現其中之一，就應該選這個關鍵字，不管畫面其他元素長什麼樣）：
- 角色手持武器（槍、刀、盾、魔法武器、科幻發射器都算）
- 軍事制服／裝甲／軍階符號
- 戰鬥姿態（瞄準、出招、衝刺攻擊）
- 戰場／戰術地圖／HUD 介面
- 電競對戰、FPS、TPS、戰爭策略遊戲的官方視覺

## 工作流程

### Step 0：判斷輸入類型

使用者可能會：
- **(A) 貼一個網址（一般網頁）** → 走 Step 1 ~ 5 完整流程
- **(B) 貼一個圖片 URL（沒有對應網頁）** → 走「純圖片分支」（B1）
- **(C) 直接附上一張圖片（拖拉、貼上、上傳本機檔案）** → 走「純圖片分支」（B2）— **這是預設情境，不要再問使用者有沒有公開 URL，直接幫他處理！**

**判斷規則**（從鬆到嚴）：
1. 對話中**附有圖片檔案** → C
2. URL 結尾是 `.png/.jpg/.jpeg/.gif/.webp/.svg/.bmp` → B
3. 不確定時用 `curl -I -L --max-time 8 -s -o /dev/null -w "%{content_type}" "<URL>"`，如果回傳 `image/*` → B
4. 否則 → A（網頁）

如果是 B 或 C，跳到本文件最下方「## 純圖片分支」段落。

### Step 0.5：判斷是否為多輸入（多張圖 / 多個 URL / 混合）

**強制規則**：以下任一情境**一律算多輸入**，**必須先問使用者要合併還是分開**，**不能自行判定**：
- 兩個或以上的網址
- 兩張或以上的圖片
- 1 個網址 + 1 張（或以上）附上的圖片 ← **這條最容易被誤判，特別注意**
- 1 個圖片 URL + 1 張附上的圖片

**唯一例外**：使用者明確說「這張圖是這個網址的截圖」「這是那個網站的內頁設計」等清楚表示**圖片來自網址**時，才當作單一輸入處理。

**禁止自作主張**：絕對不要因為「圖片看起來像那個網址的內容」「這個品牌剛好做這種風格」就自己合併。設計師同事附上的圖跟貼的網址，**常常是不同的兩個靈感來源**，要尊重他們的原始意圖。

如果偵測到多輸入，問使用者：

```
我看到你給了 X 個項目（[列出每個項目簡稱]）。你想要：

1. **合併成一張卡片** — 這些屬於同一個品牌/專案/系列，存成一張，多張圖片放進「檔案和媒體」
2. **分開建立 X 張卡片** — 每個項目獨立收藏，各自有不同的名稱、種類、評分

請回覆「合併」或「分開」。
```

依使用者選擇走不同路徑：
- **合併模式 → 走「## 合併模式」（最下方）**
- **分開模式 → 走「## 分開模式」（最下方）**

如果只有單一輸入，跳過 Step 0.5，直接走 Step 1 ~ 5（一般流程）。

### Step 1：抓取頁面資訊

**重要**：把使用者貼給你的原始網址記下來（後面 Step 4 要當 `source_url` 一起送出，讓 Notion 卡片可以點回原網站）。

用 WebFetch 或 curl 取得目標 URL 的 HTML，提取：
- `<title>` 標籤內容 → 用來命名項目
- `<meta property="og:image">` 的 content → 封面圖 URL
- `<meta name="description">` 或 `<meta property="og:description">` → 輔助分析

如果 WebFetch 拿不到完整 HTML，改用：
```bash
curl -L -A "Mozilla/5.0" --max-time 15 -s "URL" | grep -E "(og:image|og:title|og:description|<title)"
```

### Step 2：分析並填入欄位

根據頁面內容判斷：

**項目名稱**：使用網站名稱或品牌名，簡潔有力（不要貼完整標題，取關鍵詞）

**項目種類**：
- 字體、字型、排版工具 → `標準字`
- 網站 UI、App 介面、設計系統 → `UI / UX`
- 品牌 LOGO、海報、印刷品 → `平面設計`
- 動態影片、動畫 → `影片`
- 插圖、手繪、漫畫 → `插畫`
- AI 工具、Prompt 範例 → `AI Prompt`

**推薦指數**：**由使用者決定，不要自己填**。
- 你可以提出建議，但**不能直接在預覽中填上星數**
- 在預覽顯示前，先問使用者：「你想給這個幾顆星？（1-5 顆，可以說『建議幾顆』讓我提議）」
- 如果使用者說「你建議」或「幫我評」，再依下方標準提建議，並請使用者確認或調整：
  - ⭐⭐⭐⭐⭐：業界教科書級、全球知名品牌、設計被廣泛引用
  - ⭐⭐⭐⭐：專業品質高、有明確設計語言、值得參考但非標竿
  - ⭐⭐⭐：有亮點但不全面，或概念好但執行普通
  - ⭐⭐：有可學的元素但整體偏普通
  - ⭐：有點意思但參考價值低
- 拿到使用者回答後（例如「4顆」「⭐⭐⭐⭐」「四顆星」），轉成對應的 ⭐ 符號字串填入

**關鍵字**：**最多 2 個**，依「關鍵字判定標準」對照視覺特徵選。寧缺勿濫，只有 1 個準就只選 1 個。

**說明文字**：100-200 字的品牌視覺分析，用繁體中文，說明：
- 設計特色是什麼
- 用了哪些視覺語言
- 對設計師有什麼參考價值

**封面圖 URL**：
- **規則：只要從網址抓得到任何一張有效圖片，就一定要設定 `image_url`**（GAS 會自動把它寫進 Notion 的 `cover` 屬性，讓卡片頂端有大圖）
- 優先用 `og:image`
- 如果 og:image 是相對路徑（例如 `/images/og.png`），記得展開成絕對 URL
- 後備順序：`og:image` → `twitter:image` → `apple-touch-icon` → `/favicon.ico`
- 在 Step 2.5 驗證後，**只有當全部都失敗才設為空字串 `""`**（不要為了求快直接放棄）

### Step 2.5：驗證封面圖 URL 可用

**重要**：og:image 經常會壞掉（404、相對路徑沒展開、被擋 hotlink），如果不驗證就送出去，Notion 雖然會建卡片成功，但封面圖會空白或破圖。

用 `curl -I` 檢查每個候選 URL，直到找到一個有效的：

```bash
curl -I -L -A "Mozilla/5.0" --max-time 8 -s -o /dev/null \
  -w "%{http_code} %{content_type}" \
  "<候選圖片 URL>"
```

**判斷條件**（兩個都要滿足才算 OK）：
- HTTP code 是 `200`
- content_type 開頭是 `image/`（例如 `image/png`、`image/jpeg`、`image/webp`、`image/svg+xml`）

**驗證流程**：
1. 試 og:image — 通過就用它
2. 不通過 → 試 twitter:image
3. 不通過 → 試 apple-touch-icon（通常在 `<link rel="apple-touch-icon" href="...">`）
4. 不通過 → 試 `<原網域>/favicon.ico`
5. 全部都失敗 → `image_url = ""`，並在預覽中提示使用者「⚠️ 沒抓到有效的封面圖，這張卡片不會有封面」

**常見陷阱**：
- Apple、Instagram、部分公司網站會擋外部 hotlink，圖片實際存在但 curl 拿不到。這種情況把它當失敗處理沒關係，反正 Notion 也會抓不到
- 相對路徑（`/images/...`）一定要先拼成絕對 URL 再驗證
- 有些 og:image 會 301/302 跳到 CDN，記得加 `-L` 跟隨 redirect

### Step 3：給使用者預覽

**一定要先預覽，不要直接送出。**

**進入預覽前，要先取得「推薦指數」**（見 Step 2 的說明）。如果使用者還沒給星數，預覽**不能顯示**，要先問：「你想給這個網址幾顆星？（1-5 顆，或說『建議幾顆』讓我提議）」拿到答案後再顯示完整預覽。

用這個固定格式顯示預覽：

```
## 靈感卡片預覽

| 欄位 | 內容 |
|------|------|
| **項目名稱** | [名稱] |
| **項目種類** | [種類] |
| **推薦指數** | [星級] |
| **關鍵字** | `關鍵字1` `關鍵字2` `關鍵字3` |
| **說明文字** | [200字以內的分析] |
| **封面圖** | [URL 或「⚠️ 沒抓到有效封面圖，這張卡片不會有封面」] |

確認沒問題的話說「存入」，我就幫你寫入 Notion 靈感板！

如果使用者對封面圖不滿意（或想改），可以提供：
- 「請改用這張：[新的圖片網址]」
- 「不要圖片」 → image_url 設成 ""
```

### Step 4：發送資料到 GAS 中繼站
🚨 **GAS 回應的特殊行為（一定要看完，避免重試造成 Notion 重複卡片）**：

Google Apps Script 的 web app 永遠會回傳 **302 redirect → script.googleusercontent.com → 真正的 JSON**。這是 GAS 的**正常設計**，不是錯誤。

**處理方式**：
- ✅ 只用下面的 Python + curl 程式碼，不要用 WebFetch 或其他工具呼叫 GAS
- ✅ `curl -s -L` 會自動跟隨 redirect，把最終 JSON 放在 `result.stdout`
- ✅ **看 `result.stdout` 內容是否是 JSON 且 `status == "success"`，這才是判斷依據**
- ❌ **絕對不要看到 302/redirect/HTML 就重試**（會在 Notion 建出重複卡）
- ❌ **不要加 `-X POST`**（會強迫 redirect 也用 POST，導致真的 404）
- ❌ 不要用 WebFetch（它對 POST 跟 redirect 的處理不可預期）

**只有以下情況才算失敗**：
1. `result.returncode != 0`（curl 本身執行錯誤）
2. `result.stdout` 不是有效 JSON（網路完全中斷或 GAS 真的死了）
3. JSON 內 `status == "error"`

其他任何情況（包含看到 redirect 字樣、HTML 內容混在 stdout 開頭）都**不要重試**，直接回報結果。


使用者確認後（且已依「密鑰處理」章節拿到密鑰），用 Python 將資料發送給 Webhook：

```python
import subprocess, json, os, sys

# GAS 中繼站網址（寫死在 Skill 內，所有同事共用）
GAS_URL = "https://script.google.com/macros/s/AKfycbxjQmdn90grsatP85yr9IhtZUrRgpG-IjbeEew5O8k6FrWVmqXrZPUzpDGao4thc0oK/exec"

# 密鑰：優先從環境變數讀；若沒有，會在前一步已透過對話跟使用者拿到並暫存於變數中
gas_secret = os.environ.get("NOTION_GAS_SECRET", "")
# 如果環境變數沒有，請在這行替換成從對話中拿到的密鑰
# gas_secret = "（使用者剛剛輸入的密鑰）"

if not gas_secret:
    print("ERROR: 沒有密鑰，無法送出。請先詢問使用者。")
    sys.exit(1)

# 替換為實際分析後的內容
# ⚠️ 關鍵字一定要逐字比對選項清單！斜線前後有半形空格，
#    若省略空格 Notion 會在 DB 新建重複選項變成髒資料。
payload = {
    "secret": gas_secret,
    "title": "項目名稱",
    "source_url": "https://www.example.com/",   # 使用者貼給你的原始網址（原封不動帶上去）
    "category": "UI / UX",            # 必須完全符合選項列表（斜線前後有空格）
    "rating": "⭐⭐⭐⭐⭐",
    "keywords": ["科技 / 現代", "極簡 / 都會"],   # 注意斜線前後空格
    "description": "說明文字...",
    "image_url": "https://example.com/og.jpg"  # 沒有圖片時設為 ""
}

body_json = json.dumps(payload, ensure_ascii=False)

# 注意：不要加 -X POST！curl 會強迫連 GAS 的 302 redirect 也用 POST，
# 導致請求失敗。只用 -d 即可（自動 POST 第一個請求、GET 跟隨 redirect）。
result = subprocess.run(
    ["curl", "-s", "-L",
     GAS_URL,
     "-H", "Content-Type: application/json",
     "-d", body_json],
    capture_output=True, text=True
)

try:
    resp = json.loads(result.stdout)
    if resp.get("status") == "success":
        page_url = resp.get("page_url", "")
        print(f"SUCCESS: {page_url}")
    else:
        print(f"ERROR: {resp.get('message', '未知錯誤')}")
        sys.exit(1)
except json.JSONDecodeError:
    print(f"ERROR: GAS 回傳格式異常 - {result.stdout[:200]}")
    sys.exit(1)
```

**重要**：
- `category` 的值必須與選項列表完全一致（含空格），例如 `UI / UX` 前後有空格
- `keywords` 只能用選項列表中的值，不能自創新標籤
- `rating` 使用 ⭐ 符號，例如 `⭐⭐⭐⭐`
- `secret` 必須帶上，否則 GAS 會拒絕請求

**[項目名稱]**
[Notion 頁面連結]

卡片已建立，可以在 Ideas Pool 的 Gallery view 看到封面圖。

## 常見錯誤排查

**「Unauthorized」或「secret invalid」**：`NOTION_GAS_SECRET` 錯誤，向管理員確認。

**GAS 回傳 HTML 而非 JSON**：通常是 GAS 部署設定問題，請管理員確認部署為「網頁應用程式」、執行身分「我」、存取權限「所有人」。

**圖片在 Gallery view 不顯示**：管理員端 GAS 程式碼應同時設定 `cover` 與 `檔案和媒體`，若沒有請更新 GAS。

**Notion 出現重複的關鍵字選項**：表示 Skill 送出時關鍵字字串和 Notion 預設選項不完全一致（最常見：斜線前後少了空格、或漏掉「AI 相關」中間的空格）。Notion 對 multi_select 的策略是「找不到就新建」，所以一個小字差就會多一個選項。對照本 Skill 的關鍵字清單逐字檢查；若已經產生髒資料，到 Notion DB 點關鍵字欄位 → 編輯選項 → 把重複的選項刪掉。

## 純圖片分支（Step 0 判定為 B 或 C 時走這條）

當使用者沒給網頁（B：純圖片 URL / C：直接附檔），抓不到 og:title、og:description，**無法自動抓網頁分析**。改為「Claude 看圖分析 + 人工確認」的流程。

### Step P0：取得圖片資料

**重要**：GAS 中繼站支援兩種圖片傳送方式，請依輸入類型選擇對應作法：

**B 路徑（使用者貼了圖片 URL）**：
驗證該 URL 有效後，當成 `image_url` 傳給 GAS。
```bash
curl -I -L -A "Mozilla/5.0" --max-time 8 -s -o /dev/null \
  -w "%{http_code} %{content_type}" \
  "<圖片 URL>"
```
- HTTP 200 + content_type 開頭 `image/` → 通過
- 失敗 → 告訴使用者「這個圖片 URL 抓不到（HTTP XXX），請換一個」並停止流程

**C 路徑（使用者直接附上本機圖片）**：
**預設行為：把圖片讀成 base64 直接送 GAS，由 GAS 上傳到 Notion 永久存放。不要問使用者「有沒有公開網址」、不要試圖找圖床、不要用 curl 上傳外部服務。**

Notion 的 file_upload API 會把檔案存進 Notion 自家儲存空間，永久有效。流程：

```python
import base64
with open("<本機圖片絕對路徑>", "rb") as f:
    image_b64 = base64.b64encode(f.read()).decode('ascii')

# 在 Step P4 送 GAS 時，payload 多帶這三個欄位：
# "image_base64": image_b64,
# "image_filename": "原始檔名.png",
# "image_mime": "image/png"  (依實際格式: image/jpeg, image/gif, image/webp ...)
```

**找到本機圖片路徑的方法**：

**先依序執行以下搜尋（不要省略步驟，找到就停）**：

1. **檢查對話內容**：使用者訊息或系統訊息裡是否有直接寫出檔案路徑（例如 `/Users/.../image.png`），有的話直接用。

2. **搜尋最近的下載與桌面**：
   ```bash
   ls -lt ~/Downloads/ 2>/dev/null | head -10
   ls -lt ~/Desktop/ 2>/dev/null | head -10
   ```
   找最近 10 分鐘內出現的 `.png/.jpg/.jpeg/.webp/.gif` 檔。

3. **搜尋 Claude Code 暫存**（macOS）：
   ```bash
   find /var/folders -type f \( -name "*.png" -o -name "*.jpg" -o -name "*.jpeg" -o -name "*.webp" \) -mmin -15 2>/dev/null | head -10
   find ~/Library/Application\ Support/Claude -type f \( -name "*.png" -o -name "*.jpg" -o -name "*.jpeg" -o -name "*.webp" \) -mmin -15 2>/dev/null | head -10
   ```

4. **搜尋系統暫存**：
   ```bash
   find /tmp -type f \( -name "*.png" -o -name "*.jpg" -o -name "*.jpeg" -o -name "*.webp" \) -mmin -15 2>/dev/null | head -10
   ```

5. **找到候選檔案後，用 Read 工具實際看一張，確認確實是使用者剛附的圖**（避免拿到其他無關的最近檔案）。

**全部都找不到才問使用者**，且要給具體、好操作的引導，不要只說「請告訴我路徑」：

```
我搜過了 ~/Downloads、~/Desktop、Claude 暫存資料夾跟系統暫存，都沒找到剛剛附上的圖。

最簡單的解法：請把圖片**拖到桌面**（隨便存成什麼名字都行），然後告訴我「在桌面上」即可，我會自動找到最新的那張。

或者：
- 直接告訴我絕對路徑（例如 /Users/你/Downloads/cat.png）
- 改貼一個公開的圖片 URL
```

之後使用者說「在桌面上」時，直接執行 `ls -lt ~/Desktop/ | head -5` 拿最新一張。

**絕對不要編造路徑或假裝圖片在某處**。

base64 字串可能很大（一張 1MB 的圖會變成約 1.3MB base64 字串），但 GAS 可以處理 20MB 內的請求，一般圖片都沒問題。

### Step P2：分析圖片並預填欄位（不要問「有沒有公開網址」）

**重要**：到這一步圖片公開 URL 已經處理好了（Step P0 上傳完了），不要再問使用者圖片相關的事。直接：

1. 看圖（你可以實際看圖內容）→ 預填項目名稱、項目種類、關鍵字、說明文字的**建議**
2. 把建議列給使用者，請他確認或修改
3. **推薦指數**單獨問（規則：由使用者決定，不要自己填）

格式範例：

```
我看了這張圖，先幫你預填幾個欄位：

| 欄位 | 我的建議 |
|------|---------|
| **項目名稱** | （根據圖片內容給建議） |
| **項目種類** | （從六個選項挑一個） |
| **關鍵字** | `關鍵字 1`（依判定標準最多 2 個，只有 1 個準就 1 個） |
| **說明文字** | （根據圖片視覺風格寫 100-200 字分析） |

需要你決定：
1. **推薦指數**：1-5 顆星？（或說「建議」讓我幫你評）
2. 上述欄位有要修改的嗎？（例如「項目種類改 XX」「關鍵字拿掉 XX」）
```

如果使用者只想改某幾項，更新後重新顯示完整預覽。如果他說「都好」「OK」，直接進 Step P3 預覽。

### Step P3：預覽

跟一般流程的 Step 3 一樣，但因為沒有 source_url，預覽中要在「封面圖」那欄附註：
- 「✅ 圖片已上傳到 catbox.moe 永久連結，封面圖會正常顯示」
- 「（純圖片入口，沒有原始網址）」

確認沒問題說「存入」就送出。

### Step P4：送 GAS

用一般流程的 Step 4 Python 程式碼，但 `source_url` 留空字串：

```python
payload = {
    "secret": gas_secret,
    "title": "（使用者提供）",
    "source_url": "",                  # 純圖片入口沒有原始網址
    "category": "（使用者提供）",
    "rating": "（使用者提供，⭐符號）",
    "keywords": [...],
    "description": "（使用者提供）",
    # 依照 Step P0 選擇傳一組：
    # B 路徑（圖片 URL）：
    "image_url": "https://...",
    # C 路徑（本機附檔，base64 直傳）：
    "image_base64": image_b64,
    "image_filename": "image.png",
    "image_mime": "image/png"
}
```

**注意**：B 路徑和 C 路徑**不要同時送**，GAS 會優先處理 `image_base64`，外部 URL 會被忽略。

GAS 那邊會自動處理 source_url 為空的情境（不會建立「🔗 原始網址」項目）；也會自動判斷圖片來源（base64 → 上傳到 Notion；URL → 用 external type）。

### Step P5：回報

跟一般流程的 Step 5 一樣顯示成功訊息與 Notion 連結。
